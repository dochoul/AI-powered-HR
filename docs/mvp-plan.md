## 1. MVP 목표 및 범위 정의

### 1.1 MVP 목표

**핵심 가치 명제**: 경영진이 슬랙 DM에서 자연어로 직원의 기본 인사정보를 즉시 조회할 수 있다.

**성공 기준**:
- 경영진이 직원 이름으로 기본 인사정보를 3초 이내에 조회할 수 있다
- 자연어 질문의 의도를 90% 이상 정확하게 파악한다
- 모든 조회 이력이 감사 로그에 기록된다
- 권한이 없는 정보(주민번호, 계좌번호 등 극비 등급)에 대한 접근이 100% 차단된다
- 기밀 등급 정보(연봉)는 경영진/HR 담당자만 조회 가능하며, 비인가 접근이 100% 차단된다

### 1.2 MVP 지원 질문 유형

MVP에서 지원하는 질문 유형은 **직원 기본정보 조회** 단일 도메인으로 한정한다.

| 번호 | 질문 유형 | 예시 질문 | 조회 필드 |
|------|----------|----------|----------|
| 1 | 특정 직원 기본정보 전체 조회 | "김민수 정보 알려줘" | 부서, 직급, 입사일, 연락처, 이메일 |
| 2 | 특정 직원 부서 조회 | "박지영이 어느 부서야?" | 부서 |
| 3 | 특정 직원 직급 조회 | "이승호 직급이 뭐야?" | 직급 |
| 4 | 특정 직원 입사일 조회 | "최유진 입사일이 언제야?" | 입사일 |
| 5 | 특정 직원 연락처 조회 | "정하늘 연락처 알려줘" | 전화번호, 이메일 |
| 6 | 특정 부서 소속 직원 조회 | "개발팀에 누가 있어?" | 부서원 목록 (이름, 직급) |
| 7 | 특정 직급의 직원 조회 | "과장급 직원 누구누구야?" | 해당 직급 직원 목록 |
| 8 | 동명이인 구분 | "김민수가 두 명인데 어떤 김민수?" | 동명이인 목록 제시 후 선택 |
| 9 | 특정 직원 재직 상태 확인 | "한소희 아직 재직중이야?" | 재직 상태 |
| 10 | 특정 직원 근속년수 조회 | "김민수 몇 년차야?" | 입사일 기반 계산 |
| 11 | 부서별 인원수 조회 | "개발팀 몇 명이야?" | 부서별 인원 카운트 |
| 12 | 최근 입사자 조회 | "이번 달 입사한 사람 있어?" | 기간 내 입사자 목록 |
| 13 | 특정 직원 연봉 조회 | "홍길동 연봉 얼마야?" | 연봉 (기밀 등급) |
| 14 | 부서별 평균 연봉 조회 | "마케팅팀 평균 연봉은?" | 부서 평균 연봉 (기밀 등급) |
| 15 | 직급별 평균 연봉 조회 | "과장급 평균 연봉이 어떻게 돼?" | 직급별 평균 연봉 (기밀 등급) |

### 1.3 MVP 조회 가능 인사정보 필드

보안 등급에 따라 MVP에서 조회 가능한 필드를 분류한다.

**[일반 등급] - MVP에서 조회 가능**
| 필드명 | 설명 | 비고 |
|--------|------|------|
| 이름 | 직원 성명 | 검색 키 |
| 부서 | 소속 부서명 | |
| 직급 | 현재 직급 | 사원/대리/과장/차장/부장/이사/상무/전무/부사장/사장 |
| 직책 | 현재 직책 | 팀원/팀장/실장/본부장 등 (있는 경우) |
| 입사일 | 입사 연월일 | 근속년수 계산 기반 |

| 필드명 | 설명 | 비고 |
|--------|------|------|
| 휴대전화 | 개인 연락처 | |
| 생년월일 | 생년월일 | 나이 계산 기반 |

**[기밀 등급] - MVP에서 경영진/HR 담당자에게 조회 가능**
| 필드명 | 설명 | 비고 |
|--------|------|------|
| 연봉 | 연간 급여 총액 | 경영진/HR 담당자만 조회 가능, 감사 로그 필수 기록 |

**[기밀 등급] - MVP에서 조회 불가 (접근 시 안내 메시지)**
| 필드명 | 설명 | MVP 제외 사유 |
|--------|------|--------------|
| 급여 명세서 | 월별 급여 상세 내역 (기본급, 수당, 공제 등) | 급여 시스템 연동 필요 |
| 인사평가 | 평가 등급/이력 | 평가 시스템 연동 필요 |
| 근태 | 출퇴근/휴가 | 근태 시스템 연동 필요 |

**[극비 등급] - 시스템 원천 차단**
| 필드명 | 설명 | 비고 |
|--------|------|------|
| 주민등록번호 | | DB 필드 자체를 API에 노출하지 않음 |
| 계좌번호 | | DB 필드 자체를 API에 노출하지 않음 |

| 기능 | 제외 사유 | 예정 Phase |

## 2. 사용자 시나리오 (User Flow)

### 2.1 전체 플로우 개요

```
[경영진] --슬랙 DM--> [슬랙봇]
    |
    v
[슬랙봇] --메시지 전달--> [AI 처리 서버 (FastAPI)]
    |
    v
[사용자 인증/권한 확인] -- 실패 --> [권한 부족 응답]
    |
    v (성공)
[Claude API: 의도 분석 + Function Calling 결정]
    |
    v
[내부 API 호출: HR DB 조회]
    |
    v
[Claude API: 조회 결과를 자연어 응답으로 변환]
    |
    v
[감사 로그 기록]
    |
    v
[슬랙봇] --응답--> [경영진]
```

### 2.2 시나리오 1: 정상 케이스 - 직원 기본정보 조회

**전제**: 경영진 A가 슬랙에서 챗봇과 DM 채널을 열어둔 상태

```
1. 경영진 A가 슬랙 DM에 입력: "김민수 정보 알려줘"
2. 슬랙봇이 메시지를 수신하여 FastAPI 서버로 전달
3. 서버가 슬랙 사용자 ID로 권한 확인 → 경영진 역할 확인 완료
4. Claude API에 메시지 전달 → 의도 분석: "직원 기본정보 조회", 대상: "김민수"
5. Claude가 Function Calling으로 get_employee_info(name="김민수") 호출 결정
6. 서버가 HR DB에서 김민수 정보 조회 → 결과 1건
7. 조회 결과를 Claude에게 전달 → 자연어 응답 생성
8. 감사 로그 기록: {who: 경영진A, what: 직원조회, target: 김민수, when: timestamp}
9. 슬랙 DM으로 응답 전송
```

**응답 예시**:
> 김민수님의 정보입니다.
>
> - 부서: 개발팀
> - 직급: 과장
> - 직책: 백엔드파트장
> - 입사일: 2020-03-15 (6년차)
> - 이메일: minsu.kim@a-company.com
> - 연락처: 010-1234-5678

### 2.3 시나리오 2: 동명이인 처리

```
1. 경영진 A: "김민수 부서가 어디야?"
2. DB 조회 결과 김민수 2명 존재
3. Claude가 동명이인 확인 응답 생성
```

**응답 예시**:
> 김민수님이 2명 있습니다. 어떤 분을 찾으시나요?
>
> 1. 김민수 - 개발팀 과장
> 2. 김민수 - 영업팀 대리

```
4. 경영진 A: "개발팀"
5. 서버가 이전 대화 컨텍스트(동명이인 목록)를 참조하여 개발팀 김민수 확정
   (MVP에서는 직전 1턴의 컨텍스트만 유지)
6. 개발팀 김민수 정보 응답
```

**주의**: MVP에서 멀티턴은 지원하지 않지만, 동명이인 해소를 위한 **직전 1턴 컨텍스트**는 예외적으로 유지한다. 서버 세션에 직전 응답의 동명이인 목록을 임시 저장(TTL 5분)하는 방식으로 구현한다.

### 2.4 시나리오 3: 조회 결과 없음

```
1. 경영진 A: "홍길동 정보 알려줘"
2. DB 조회 결과 0건
```

**응답 예시**:
> 홍길동님을 찾을 수 없습니다. 이름을 다시 확인해 주세요.
> 혹시 비슷한 이름을 찾으시나요?
>
> - 홍길순 (마케팅팀 대리)
> - 홍길만 (경영지원팀 사원)

**구현 참고**: 유사 이름 검색은 DB의 LIKE 검색 또는 초성 검색으로 구현한다. 유사 결과가 없으면 단순 안내 메시지만 표시한다.

### 2.5 시나리오 4: 권한 부족 (비인가 사용자)

```
1. 일반 직원 B가 슬랙 DM에 입력: "김민수 정보 알려줘"
2. 서버가 슬랙 사용자 ID로 권한 확인 → 경영진 역할 아님
3. 즉시 차단 (Claude API 호출 없이)
```

**응답 예시**:
> 죄송합니다. 인사정보 조회 권한이 없습니다.
> 필요하시면 HR팀에 문의해 주세요.

### 2.6 시나리오 5: 연봉 조회 - 정상 케이스 (경영진)

**전제**: 경영진 A가 슬랙에서 챗봇과 DM 채널을 열어둔 상태

```
1. 경영진 A: "김민수 연봉이 얼마야?"
2. 서버가 슬랙 사용자 ID로 권한 확인 → 경영진 역할 확인 완료
3. Claude가 의도 분석 → "연봉 조회", 대상: "김민수"
4. Claude가 Function Calling으로 get_employee_salary(name="김민수") 호출 결정
5. 서버가 요청자의 기밀 등급 접근 권한 확인 → 경영진이므로 허용
6. HR DB에서 김민수 연봉 정보 조회
7. 감사 로그 기록: {who: 경영진A, what: 연봉조회, target: 김민수, when: timestamp}
8. 응답 전송
```

**응답 예시**:
> 김민수님의 연봉 정보입니다.
>
> - 이름: 김민수 (개발팀 과장)
> - 연봉: 65,000,000원
>
> _기밀 정보이므로 조회 이력이 기록됩니다._

### 2.6.1 시나리오 5-1: 연봉 조회 - 권한 부족 (일반 직원/팀장)

```
1. 팀장 B가 슬랙 DM에 입력: "김민수 연봉이 얼마야?"
   (Phase 3에서 팀장이 챗봇 접근 가능해진 이후 시나리오)
2. 서버가 슬랙 사용자 ID로 권한 확인 → 팀장 역할
3. 팀장은 기본정보 조회 가능하나, 기밀 등급(연봉) 접근 불가
4. 감사 로그 기록: {who: 팀장B, what: 연봉조회시도, target: 김민수, result: unauthorized}
```

**응답 예시**:
> 죄송합니다. 연봉 정보는 조회 권한이 없습니다.
> 급여 관련 문의는 HR팀에 직접 문의해 주세요.

### 2.6.2 시나리오 5-2: 부서 평균 연봉 조회 (경영진)

```
1. 경영진 A: "마케팅팀 평균 연봉은?"
2. 서버가 권한 확인 → 경영진, 기밀 등급 접근 허용
3. Claude가 Function Calling으로 get_department_salary_stats(department="마케팅팀") 호출
4. HR DB에서 마케팅팀 연봉 통계 조회
5. 감사 로그 기록
6. 응답 전송
```

**응답 예시**:
> 마케팅팀의 연봉 현황입니다.
>
> - 인원: 5명
> - 평균 연봉: 52,000,000원
> - 최고: 78,000,000원
> - 최저: 35,000,000원
>
> _기밀 정보이므로 조회 이력이 기록됩니다._

### 2.6.3 시나리오 5-3: MVP 범위 밖 질문 (급여 명세서 등)

```
1. 경영진 A: "김민수 이번 달 급여 명세서 보여줘"
2. Claude가 의도 분석 → "급여 명세서 조회" → MVP 범위 밖
```

**응답 예시**:
> 급여 명세서는 현재 조회할 수 없습니다. 해당 기능은 추후 업데이트 예정입니다.
> 연봉 정보가 필요하시면 "김민수 연봉 알려줘"로 질문해 주세요.
> 급여 상세 내역은 HR팀에 직접 문의해 주세요.

### 2.7 시나리오 6: 의도 파악 불가

```
1. 경영진 A: "오늘 날씨 어때?"
2. Claude가 의도 분석 → HR 관련 질문 아님
```

**응답 예시**:
> 저는 인사정보 조회를 도와드리는 챗봇입니다.

### 2.8 시나리오 7: 부서 단위 조회

**권한 검증 흐름**:
```
1. 슬랙 메시지 수신
2. 슬랙 사용자 ID 추출
3. user_slack_mapping 테이블에서 내부 employee_id 조회
   → 매핑 없음: "등록되지 않은 사용자" 응답
4. employee_role 테이블에서 역할 확인
   → 경영진 아님: "권한 없음" 응답
5. 권한 확인 완료 → Claude API 호출 진행
6. [기밀 등급 조회 시 추가 검증]
   Claude가 get_employee_salary 또는 get_department_salary_stats 호출 결정 시,
   서버가 요청자의 역할이 executive 또는 hr_admin인지 재확인
   → 해당 역할 아님: "연봉 정보 조회 권한 없음" 응답
   → 해당 역할 맞음: DB 조회 실행 + 감사 로그에 기밀 등급 조회 기록
```

### 7.3 API 엔드포인트

| Method | Path | 설명 | 인증 |
|--------|------|------|------|
| POST | `/api/slack/events` | 슬랙 이벤트 수신 (Event API) | 슬랙 Signing Secret 검증 |
| GET | `/health` | 서버 헬스체크 | 없음 |
| GET | `/health/db` | DB 연결 상태 확인 | 없음 |

**참고**: MVP에서는 슬랙 Event API를 통한 단일 엔드포인트만 존재한다. REST API로 직접 조회하는 기능은 Phase 3(관리자 대시보드)에서 추가한다.

#### POST /api/slack/events

**요청 (슬랙 Event API)**:
```json
{
  "type": "event_callback",
  "event": {
    "type": "message",
    "channel_type": "im",
    "user": "U0123ABCDEF",
    "text": "김민수 정보 알려줘",
    "channel": "D0123GHIJKL",
    "ts": "1710230400.000100"
  }
}
```

**처리 흐름**:
1. 슬랙 Signing Secret으로 요청 검증
2. `channel_type`이 `im`이 아니면 무시
3. 봇 자신의 메시지면 무시 (무한 루프 방지)
4. 비동기로 처리 시작, 슬랙에 즉시 200 OK 반환 (3초 룰)
5. 내부 처리 완료 후 슬랙 Chat API로 응답 발신

## 부록 A: MVP 체크리스트

기능 출시 전 최종 확인 항목:

- [ ] 경영진이 슬랙 DM에서 직원 이름으로 기본 정보를 조회할 수 있다
- [ ] 경영진이 부서별/직급별 직원 목록을 조회할 수 있다
- [ ] 동명이인이 있을 때 선택 요청이 정상적으로 동작한다
- [ ] 비인가 사용자의 접근이 Claude API 호출 전에 차단된다
- [ ] 극비 정보(주민번호, 계좌번호)가 어떤 경우에도 노출되지 않는다
- [ ] 경영진이 직원 연봉 및 부서별 연봉 통계를 조회할 수 있다
- [ ] 연봉 조회 시 기밀 등급 권한이 서버에서 재확인된다
- [ ] 연봉 조회 이력이 보안 등급과 함께 감사 로그에 기록된다
- [ ] 기밀 정보(급여 명세서, 평가) 요청 시 적절한 안내 메시지가 표시된다
- [ ] 모든 조회 요청이 감사 로그에 기록된다
- [ ] HR과 무관한 질문에 대해 적절한 안내가 제공된다
- [ ] 검색 결과가 없을 때 유사 이름이 제안된다
- [ ] 전체 응답 시간이 5초 이내이다
- [ ] API 키와 토큰이 환경변수로 안전하게 관리된다
- [ ] 채널 메시지에는 응답하지 않는다 (DM 전용)
